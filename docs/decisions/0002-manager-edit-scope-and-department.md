# 0002: 管理職の編集範囲と部署マスタの追加

## 状況

要件定義書3.1で「管理職が編集できる他社員の範囲は全社員か自部署配下のみか」が未確定❓だった。全社員editable案はデータモデルへの追加が不要でMVP向けにシンプルだが、実際の業務要件（部長は自部署のメンバーの経歴書のみ触れる想定）とは乖離があるという指摘がユーザーからあった。

## 決定

- 管理職（MANAGER）が編集できる他社員の範囲は**自部署配下のみ**とする
- 判定は部署単位の粗い比較（`manager.departmentId === target.departmentId`）とし、個人ごとの上長関係（誰が誰の直属の部下か）は持たない
- このため`Department`（部署マスタ）と`Employee.departmentId`をデータモデルに追加し、フェーズ1スコープに組み込む

## 比較・理由

| 観点 | 全社員編集可（旧案） | 自部署配下のみ（採用） |
|---|---|---|
| 実装コスト | 低い（追加テーブル不要） | 中（Department追加、権限判定ロジック追加） |
| 業務実態との整合性 | 低い（部長が他部署の経歴書まで編集できてしまう） | 高い |
| MVPとしての適切さ | 権限機能のデモとしては物足りない | 権限マトリクスの意義を示せる、ハッカソンでのデモ映えも良い |

個人単位の上長階層まで作ると`ManagerRelation`のような追加テーブルや異動履歴管理まで必要になり、ハッカソンの期間に対して過剰と判断。部署単位の粗い権限に留めるのが妥当という結論になった。

## 影響

- `docs/design/data-model.md`: `Department`テーブル追加、`Employee.departmentId`追加、権限判定ロジックの節を追加
- `docs/design/screens.md`: S-18（部署マスタ管理画面）、S-19（他社員プロフィール編集画面）を追加
- `docs/requirements/requirements.md`: 3.1・3.4・フェーズ表を更新
- 今後、部署の階層化や個人単位の上長関係が必要になった場合は、`parentDepartmentId`や`managerId`の追加で拡張する（本ADRでは見送り）

## 補足（2026-07-08更新）

- 組織は「事業部＞部署＞Gr」の3階層に拡張された（ADR `0007-screen-list-pdf-adoption.md`）。これに伴い、本ADRの判定式`manager.departmentId === target.departmentId`（完全一致）は「所属組織を部署（DEPARTMENT）レベルに解決したうえでの一致」に更新された
- また、閲覧権限にも部署単位の制限（一般ユーザーは自部署のみ詳細閲覧可）が追加された。詳細はADR `0008-view-permission-department-scope.md`を参照。「管理職の編集範囲は自部署のみ」という本ADRの決定自体は維持されている
