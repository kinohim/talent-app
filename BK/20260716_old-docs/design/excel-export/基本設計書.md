## 01_システム構成

| 項目 | 内容 |
| --- | --- |
| アーキテクチャ概要 | ブラウザ→HTTPS→Next.jsアプリ（フロント+API Routesを1リポジトリで提供、ページ/UIコンポーネント、middleware.tsで認証・権限の一次チェック、API Routes、Auth.jsのCredentials Provider）→Prisma Client→PostgreSQL（Neon or Supabase） |
| デプロイ方針 | フロントとAPIを1つのNext.jsアプリに統合し、デプロイ先はVercel等のサーバーレス環境を想定（ADR 0001-tech-stack.md） |
| DB方針 | DBはマネージドPostgreSQL。ローカル開発時も同一のPostgreSQLに接続する構成とし、SQLiteとの差異による不具合を避ける |
| 未確定事項 | パスワードリセットメールの送信手段（メールサービスの選定）は未確定。フェーズ2着手時に決定する（候補: Resend, SendGrid等の無料枠） |

## 05_データモデル

| テーブル名 | 主なカラム | 説明 |
| --- | --- | --- |
| User（認証） | id, employeeId(FK→Employee,UNIQUE), username, passwordHash, role(GENERAL/MANAGER/ADMIN), email, lastLoginAt, passwordResetToken, passwordResetExpiresAt | 認証情報。Employeeと1対1 |
| Department（部署マスタ） | id, code(UNIQUE), departmentName | 新規追加。管理職の編集範囲判定に使用。階層は持たないフラット構成 |
| Employee（経歴書） | employeeId(PK,String(6)), name, nameKana, birthDate, gender, departmentId(FK), nearestStationId(FK), experienceYears, careerSummary, selfPr, finalSchoolName, finalDepartmentName, finalSchoolType, graduationYearMonth, graduationStatus | 経歴書本体。社員IDは単純連番 |
| RailwayLine/Station/StationLineLink | id,lineName / id,stationName / id,lineId(FK),stationId(FK) | 駅・路線マスタ（多対多） |
| SkillCategory/Skill/SkillVersion | id,code,categoryName / id,categoryId(FK),skillName,hasVersion / id,skillId(FK),versionName,versionOrder,releaseDate,isActive,displayName | スキルマスタ（バージョン管理あり） |
| EmployeeSkillLink | id, employeeId(FK), skillId(FK), skillVersionId(FK,nullable), skillLevel(LOW/MID/HIGH) | 社員とスキルの中間テーブル。UNIQUE(employeeId,skillId,skillVersionId) |
| CertificationCategory/Certification | id,code,categoryName,description / id,categoryId(FK),certificationName,certificationOrganization | 資格マスタ |
| EmployeeCertificationLink | id, employeeId(FK), certificationId(FK), acquiredDate, expirationDate | 社員と資格の中間テーブル。再取得履歴のためUNIQUE制約なし |
| ProjectRole | id, roleName（SE/PG/リーダー等） | 現場ポジション（役割）マスタ |
| Project | id, employeeId(FK), siteName, projectTitle, projectSummary, startDate, endDate, role, totalTeamSize, teamSize | プロジェクト経歴本体 |
| ProjectRoleLink | id, projectId(FK), projectRoleId(FK) | プロジェクトと役割の中間テーブル |
| ProjectDetail | id, projectId(FK,UNIQUE), overview, researchAnalysis, requirementsDefinition, basicDesign, detailedDesign, development, testing, operation | プロジェクトの業務工程チェックボックス群 |
| ProjectSkillLink | id, projectId(FK), skillId(FK), skillVersionId(FK,nullable) | プロジェクトで使用したスキルの中間テーブル |

## 06_権限設計

| 機能 | GENERAL(一般社員) | MANAGER（管理職） | ADMIN（システム管理者） | 判定方法 |
| --- | --- | --- | --- | --- |
| 自分の経歴書編集 | ○ | ○ | ○ | session.employeeId === target.employeeId |
| 他社員の経歴書閲覧 | ○ | ○ | ○ | 制限なし |
| 他社員の経歴書編集 | × | ○（自部署配下のみ） | ○（全社員） | MANAGERはsession.departmentId===target.departmentId／ADMINは常にtrue |
| 横断検索 | ○ | ○ | ○ | 制限なし |
| マスタ管理 | × | × | ○ | session.role==='ADMIN' |
| アカウント管理 | × | × | ○ | session.role==='ADMIN' |

## 08_非機能要件

| 項目 | 内容 |
| --- | --- |
| セキュリティ | パスワードはハッシュ化（bcrypt想定）して保存。権限に応じた画面・API制御を必須とする。HTTPS前提 |
| 性能 | 社内利用・数十〜数百名規模を想定し特別な性能要件は設けない（ページネーション程度の配慮で十分） |
| 可用性・運用 | 本運用の可用性目標（SLA、監視、バックアップ）は今回のスコープ外。ハッカソン後に本運用へ進める場合は見直す |
| ブラウザ対応 | 社内利用のモダンブラウザ（Chrome/Edge最新版）のみ対応、レガシーブラウザは対象外 |
| アクセシビリティ | 特別な対応は今回のスコープ外（フェーズ3で検討） |
