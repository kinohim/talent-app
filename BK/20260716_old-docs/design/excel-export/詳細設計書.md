## 01_認証詳細

| 項目 | 内容 |
| --- | --- |
| パスワードハッシュ | bcrypt（cost factor 10〜12を想定）。平文パスワードはログにも出力しない |
| セッション戦略 | Auth.jsのJWTセッションを採用（DBセッションテーブルは持たない） |
| セッション格納情報 | employeeId, role（GENERAL/MANAGER/ADMIN）, departmentId |
| ログイン失敗時の表示 | 社員IDかパスワードのどちらが誤りかは区別せず「社員IDまたはパスワードが正しくありません」と表示 |
| パスワードリセット（フェーズ2） | passwordResetTokenとpasswordResetExpiresAt（発行から1時間想定）をUserに保持。使用後は即失効 |

## 02_権限チェック方針

| 項目 | 内容 |
| --- | --- |
| 二重チェック構成 | middleware.tsでセッション有無・大まかなアクセス可否を判定（UX目的）。APIハンドラで必ずrole/departmentId/employeeIdを取得し最終判定する |
| 編集可否の判定ルール | ADMINは常に許可。MANAGERは対象社員のdepartmentIdが自分と一致する場合のみ許可（departmentIdがnullの社員はMANAGERの対象外）。GENERALは自分自身のみ許可 |
| マスタ管理・アカウント管理 | ADMINのみ許可 |
| 実装配置 | 共通の認可関数（例: canEditEmployee, requireAdmin）を1つの共通モジュールに置き、全APIハンドラ・Server Actionから再利用する |

## 05_API共通仕様

| 項目 | 内容 |
| --- | --- |
| ベースパス | /api |
| 形式 | REST風（リソース単位のCRUD）。JSON送受信 |
| 認証 | Auth.js（NextAuth）のセッションCookie。未認証は401 |
| 認可 | ロール（GENERAL/MANAGER/ADMIN）と対象データの所有関係をサーバー側で必ずチェック。権限不足は403 |
| バリデーション | zodでリクエストボディ・クエリパラメータのスキーマを定義。不正時は400 |
| 論理削除 | 更新系はdeletedAt:nullのレコードのみ対象。一覧系はincludeDeleted=trueで削除済みも含められる |
| ページネーション | page（1始まり）・pageSize（デフォルト20、最大100）。レスポンスはitems, total, page, pageSizeを含む形 |
| エラーレスポンス形式 | errorオブジェクトの中にcode, message, detailsを含む形で統一 |

## 06_エラーコード

| HTTPステータス | codeの例 | 用途 |
| --- | --- | --- |
| 400 | VALIDATION_ERROR | リクエスト内容の検証エラー |
| 401 | UNAUTHENTICATED | 未ログイン |
| 403 | FORBIDDEN | 権限不足（自部署外の編集試行等） |
| 404 | NOT_FOUND | 対象データなし（論理削除済みを含む） |
| 409 | CONFLICT | 一意制約違反等（同一社員・同一スキルの重複登録など） |
| 500 | INTERNAL_ERROR | 想定外のサーバーエラー |

## 07_Excel取込仕様

| 項目 | 内容 |
| --- | --- |
| 画面 | S-20 Excel取込（/admin/import、管理者専用。docs/design/screens.md参照） |
| 対象 | 初回のみ。既存Excel台帳から本システムへのデータ移行専用 |
| ファイル形式 | Excel（.xlsx）。シート名は「社員」の1シート構成 |
| 取込方法 | システム管理者が管理画面からアップロード、またはスクリプトで直接投入。継続APIとしては設計しない |
| 想定カラム | 氏名、カナ、生年月日、性別、部署コード、最寄り駅名、経験年数、経歴概要、自己PR、保有スキル（カンマ区切り、スキル名:習熟度）、保有資格（カンマ区切り） |
| 社員ID | ファイルに含めず、取込時に単純連番で自動採番する |
| 部署コード | 事前に部署マスタへ登録済みであることが前提。未登録コードはエラー |
| 取込方式 | all-or-nothing。いずれかの行でエラーがあれば全体を中断し、行番号・理由を表示する |
| 画面表示 | 取込結果は行番号・成功/エラー・エラー理由の一覧で表示する（all-or-nothingのため、エラー時は全行分の理由を一覧表示し、成功時は取込件数のみ表示） |

## 08_エラーハンドリング方針

| 項目 | 内容 |
| --- | --- |
| インラインエラー表示 | APIエラーのdetailsがあればフォームの該当フィールド下に表示 |
| トースト通知 | detailsがない一般エラー（403/404/500等）はトーストでerror.messageを表示 |
| クライアント側チェック | zod schemaをフロント・API双方で共有してUX向上に使うが、最終判定は必ずサーバー側で行う |
