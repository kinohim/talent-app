# 画面設計

画面構成は`docs/design/excel-export/01_画面一覧.pdf`（22画面）を正とする（【決定】ADR `0007-screen-list-pdf-adoption.md`）。本書はその画面一覧にルーティング案・権限・設計上の要点を付与して整理したもの。

旧画面体系（S-01〜S-20、`docs/design/screens-review.html`のワイヤーフレーム含む）は旧仕様であり、対応関係はADR `0007`の「旧画面体系からの主な削除・変更」を参照。

## 権限の凡例

ロールは GENERAL（一般）／MANAGER（管理職）／HR_SALES（人事・営業）／ADMIN（システム管理者）の4階層（ADR `0008-view-permission-department-scope.md`）。

- **閲覧権限**: 経歴書詳細はGENERALのみ自部署（部署レベルの組織単位一致）に制限。MANAGER/HR_SALES/ADMINは全社員閲覧可
- **編集権限**: 本人＋MANAGER（自部署のみ）＋ADMIN（全社員）。HR_SALESは他社員編集不可

## 画面遷移の要点

- ログイン（AUTH001）→ Azure ADリダイレクト → トップ（REF001）。ロールに応じてメニューを出し分ける（画面自体は共通）
- 経歴書閲覧の動線: REF002（一覧・検索）→ REF003（詳細）→（権限があれば）EDT001〜005。本人はREF004（マイページ）→ EDT001〜005／REF006
- REF003・REF004 →「PDF出力」→ REF005（印刷用プレビュー）
- GENERALがREF002の検索結果から他部署の社員を選択することはできない（詳細リンク非活性）。URL直打ちした場合もサーバー側で403とする（`docs/design/detailed-design.md`「権限チェック」参照）
- アカウント管理: REF007 → EDT006（新規）／EDT007（編集・無効化）
- マスタ管理系（MST001〜005）は「一覧→新規追加／編集」の同一パターン。共通の一覧＋フォームコンポーネントを作り再利用する。削除時はCMN001（削除確認モーダル）を共通利用
- パスワード関連画面・Excel取込画面は持たない（ADR `0007`）

## 実装上の提案

- マスタ管理画面（MST001〜005）とアカウント一覧（REF007）は`AdminMasterTable`のような共通コンポーネントを1つ作り、カラム定義とAPIエンドポイントだけ差し替える設計にすると実装量を大きく減らせる。ただしMST004（部署）は階層表示（事業部＞部署＞Grのツリー）が必要なため、共通コンポーネントの拡張または専用実装とする
- EDT001〜005はいずれも「対象社員の経歴書の1セクションを編集する」画面。対象`employeeId`と編集セクションだけが異なるため、本人編集（マイページ経由）と他社員編集（MANAGER/ADMIN、REF003経由）でフォームコンポーネントを共通化する。EDT001の所属組織欄は「ADMINのみ編集可」フラグで出し分ける
- 権限による表示分岐はミドルウェア（Next.jsの`middleware.ts`）＋UIコンポーネント側の出し分けの二重チェックにする。ただし最終的な可否判定（閲覧の部署制限を含む）は必ずAPI側でも行う（詳細は`docs/design/detailed-design.md`「権限チェック」節）
- EDT005内の現場（プロジェクト）経歴は、`Project`を配列として動的に追加・削除できるリピーターUIとし、各行の展開先で`ProjectDetail`（業務工程チェックボックス群）・`ProjectRoleLink`・`ProjectSkillLink`を編集する入れ子フォームにする（項目定義は`docs/design/detailed-design.md`参照）
- REF005（印刷用プレビュー）の実現方式（ブラウザ印刷ベース／サーバーサイドPDF生成）はフェーズ2着手時に決定する（要件定義書5.2）
