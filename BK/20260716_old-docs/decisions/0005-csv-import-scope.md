# 0005: データ取込機能（Excel取込）の位置づけ

> **更新（2026-07-05）**: 当初はCSV取込として検討していたが、取込ファイル形式をExcel（.xlsx）に変更した。移行元の台帳がそもそもExcelであるため、CSV変換の手間を挟まずそのまま取り込めるようにする狙い。ファイル名は`0005-csv-import-scope.md`のまま残しているが、内容は「Excel取込」を指す

## 状況

要件定義書3.5・6章で「取込機能を継続機能として残すか、初回投入限定にするか」が未確定だった。継続機能にする場合はバリデーション・重複行・部分エラー時の扱いなど設計すべき項目が大きく増える。

## 決定

- データ取込は**初回投入限定**の機能と位置づける（Excel台帳からの移行専用）
- ファイル形式は**Excel（.xlsx）**とする（CSVへの変換は不要）
- 運用開始後の継続的な一括更新機能としては設計しない

## 比較・理由

| 観点 | 初回投入限定（採用） | 継続機能として保持 |
|---|---|---|
| 実装コスト | 低い（1回限りのバッチ/管理画面で十分） | 高い（バリデーション、重複・部分失敗のロールバック方針、権限制御が必要） |
| ハッカソンの期間との適合 | 高い | 低い（フェーズ2以降でも重い） |
| 本運用移行時の拡張余地 | 必要になれば別途設計して追加可能 | - |

本プロジェクトの主目的（AI駆動開発の学習・ハッカソン発表）と期間を踏まえ、継続的な取込機能に必要な堅牢なバリデーション・エラーハンドリング設計まで作り込むのは過剰と判断した。初回投入は使い捨てのスクリプトまたは簡易画面で対応し、以後のデータ更新はアプリのCRUD画面から行う想定とする。

## 影響

- `docs/requirements/requirements.md`のフェーズ2記載を「Excel取込（初回投入限定機能として実装）」に更新
- `docs/design/api-design.md`・`docs/design/detailed-design.md`ではExcel取込を継続APIとしては設計しない（初回投入は別ドキュメント／スクリプトとして扱う）。詳細仕様（取込項目・シート構成）は`docs/design/detailed-design.md`に記載
- 一覧の出力機能（CSV/Excel出力）は本決定の対象外。別途ADR `0006-remove-export-feature.md`で「持たない」と決定した
