# 0003: 社員IDの発番ルール

## 状況

要件定義書6章で「社員IDの発番ルール（連番か部署コード含む形式か）」が未確定❓だった。`docs/design/data-model.md`では`Employee.employeeId`が`String(6)`のPKとして既に定義されており、発番方式だけが決まっていなかった。

## 決定

- 社員IDは**単純連番・6桁ゼロ埋め**とする（例: `000001`, `000002`…）
- 部署コードを含めない

## 比較・理由

| 観点 | 単純連番（採用） | 部署コード＋連番 |
|---|---|---|
| 実装コスト | 低い（採番カウンタのみ） | 中（部署コード決定・部署変更時の扱いが必要） |
| 部署異動時の扱い | 影響なし（IDは部署に依存しない） | 異動時にIDを変えるか固定するかの追加検討が必要 |
| 可読性 | 部署が一目で分からない | 部署が一目で分かる |

社員が部署異動した際にIDを変更するかどうかという追加の意思決定が発生する「部署コード＋連番」案は、0002で導入した`departmentId`（可変）と社員IDの不変性の間で矛盾を生みやすいため見送った。部署が知りたい場合は`Employee.departmentId`を参照すればよく、IDに情報を埋め込む必要性は薄いと判断した。

## 影響

- `docs/design/data-model.md`の`Employee.employeeId`の説明を更新
- 採番は初期実装ではDB側の連番管理（例: 別テーブルでの採番カウンタ、またはPostgreSQLのシーケンス）で行う想定。詳細は実装フェーズで`docs/design/detailed-design.md`に追記する
