# 0009: 実装ブロッカー4件の解消と暫定認証方式

## 状況

ADR 0007/0008の設計改訂後、実装追従に着手する前の設計レビューで、実装をブロックする未決事項が確認された（2026-07-08、ユーザー決定事項）。

## 決定

### 1. EDT006（新規アカウント登録）の入力項目に氏名・カナを追加する

DB設計では`user.employee_id`がEmployeeへのFK（NOT NULL）かつ`employee.name`/`name_kana`もNOT NULLのため、氏名なしではアカウントを作成できない。EDT006の入力項目を「社員ID・氏名・カナ・メールアドレス（Azure AD照合キー）・所属部署・初期権限」とし、User＋Employee（最小項目）を同一トランザクションで作成する。DB制約は現状のまま変更しない。

### 2. 認証は当面「開発用ログイン」のみで実装する（Entra ID統合は後回し）

Azure ADテナント・アプリ登録が未準備のため、Entra IDプロバイダの実装は保留する。当面は開発用ログイン（登録済みユーザーをメールアドレスまたは社員IDで指定してログイン。パスワードの保存・照合は行わない）で全画面を実装する。

- 設計上の認証方式（AUTH001＝Entra IDリダイレクト）は変更しない。開発用ログインは代替実装であり、Auth.jsのプロバイダ差し替えで移行できる構造にする
- パスワード管理（ハッシュ保存・変更・リセット）はADR 0007どおり廃止する（開発用ログインでも使わない）
- Entra ID移行時の照合キーは`User.email`（EDT006で事前登録）

### 3. アカウント無効化は毎リクエストDBチェックで即時反映する

detailed-design.mdで未確定だった項目。APIリクエストごとに`User.isActive`をDB確認し、無効化されたアカウントは即時に401を返す。社内アプリの規模ではパフォーマンス影響は無視できる。

### 4. 社員IDは当面EDT006で手動入力とする（将来Entra ID連携で見直し）

将来的にはEntra IDリダイレクトログイン時にADアカウントへ紐づく社員番号を反映させる想定。実際にADに社員番号が紐づいているかは確認が必要なため、後日修正の可能性がある。当面はEDT006で管理者が手動入力する（6桁ゼロ埋めのバリデーションと重複チェックを実装）。ADR 0003の「自動採番」は本決定で上書きされる（連番フォーマット自体は維持）。

### 5. （付随決定）REF005のPDF出力はブラウザ印刷ベースとする

requirements.md 5.2で「フェーズ2着手時に決定」とされていた項目。印刷用プレビュー画面（REF005）は印刷用CSSを適用した専用レイアウトページとし、ブラウザの印刷機能（PDFに保存）でPDF出力を実現する。サーバーサイドPDF生成（puppeteer等）は依存が重くハッカソンスコープに見合わないため採用しない。

## 影響

- `docs/design/screens.md`: EDT006の項目に氏名・カナを追記。AUTH001に開発用ログインの暫定注記
- `docs/design/detailed-design.md`: 無効化反映方式を「毎リクエストDBチェック」に確定
- `docs/design/api-design.md`: `/api/accounts`はUser＋Employeeの同時作成に更新
- `docs/requirements/requirements.md`: 5.2未確定事項のうち解消された項目を更新
- 実装: 開発用ログインプロバイダ、`isActive`チェック、EDT006のUser+Employee同時作成
