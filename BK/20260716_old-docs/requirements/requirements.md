# 要件定義書（経歴書管理システム）

## 1. 背景・目的
Excelで管理している社員の経歴書情報をWebアプリ化し、アプリ上で閲覧・編集ができるようにする。

## 2. 誰の・どんな課題を解決するか
- 対象ユーザー: 一般社員、管理職（部長など、自部署の経歴書も編集できるユーザー）、人事・営業（全社員の経歴書を閲覧・検索するユーザー）、システム管理者（マスタデータ・アカウントを管理するユーザー）
- 課題: 経歴書がExcelで個別管理されており、検索・横断的な把握（資格保有者は誰か、特定スキル保有者は誰か、過去にどの現場に誰がいたか等）がしにくい
- 解決後の姿: Web上で経歴書を一元管理し、検索・横断表示ができる状態

## 3. スコープ（機能要件）

画面構成は`docs/design/excel-export/01_画面一覧.pdf`（22画面）を正とする（【決定】ADR `0007-screen-list-pdf-adoption.md`）。画面一覧・ルーティングは`docs/design/screens.md`参照。

### 3.1 認証・権限
- ログイン: Azure AD（Microsoft Entra ID）へのリダイレクトログイン（AUTH001）。未登録・退職無効化済みアカウントはエラー表示
- パスワードはAzure AD側で管理するため、本システムではパスワード保存・変更・リセット機能を持たない（ADR `0007`）
- 権限は4階層：一般社員（GENERAL）／管理職（MANAGER）／人事・営業（HR_SALES）／システム管理者（ADMIN）

| 機能 | 一般 | 管理職 | 人事・営業 | システム管理者 |
|---|---|---|---|---|
| 自分の経歴書編集 | ○ | ○ | ○ | ○ |
| 他社員の経歴書詳細閲覧 | ○（**自部署のみ**） | ○（全社員） | ○（全社員） | ○（全社員） |
| 経歴書一覧の横断検索（資格・スキル・現場等） | ○（**検索結果には他部署の社員も表示**） | ○ | ○ | ○ |
| 他社員の経歴書編集 | × | ○（自部署のみ） | × | ○（全社員） |
| スキルマップ／組織ダッシュボード閲覧 | ○ | ○ | ○ | ○ |
| 経歴書のPDF出力（印刷用プレビュー） | ○（閲覧可能な経歴書のみ） | ○ | ○ | ○ |
| マスタ管理（スキル/資格/現場ポジション/部署/現場） | × | × | × | ○ |
| アカウント管理 | × | × | × | ○ |

- 【決定】一般社員が閲覧できる他社員の経歴書詳細は**自分と同じ部署の社員のみ**とする。ただし経歴書一覧（REF002）の検索結果には他部署の社員も表示され、一覧行の情報（氏名・所属・ヒットしたスキル/資格/現場等）までは確認できる（ADR `0008-view-permission-department-scope.md`）
- 【決定】管理職が編集できる「他社員」の範囲は**自部署のみ**とする（ADR `0002-manager-edit-scope-and-department.md`）。組織が「事業部＞部署＞Gr」の3階層になったため、閲覧・編集いずれの「同じ部署」判定も**部署（DEPARTMENT）レベルの組織単位の一致**で行う（ADR `0008`）

### 3.2 経歴書（プロフィール）管理
- マイページ（REF004）: 本人が自分の経歴書のメニューを確認し、各編集画面へ遷移する起点
- 経歴書詳細（REF003）: 経歴書の全項目を表示する閲覧専用画面。他社員閲覧と本人確認の共通画面（閲覧可否は3.1の権限に従う）
- 編集はセクション単位の画面で行う: 基本情報（EDT001）／経歴概要・自己PR（EDT002）／スキル（EDT003）／資格（EDT004）／プロジェクト経歴（EDT005）
- プロジェクト経歴一覧（REF006）: 本人が担当したプロジェクト経歴の一覧（件数上限なし）
- 保持項目: 氏名、カナ、生年月日、性別、所属組織、最寄り駅、経験年数、経歴概要、自己PR、最終学歴、保有スキル（バージョン・習熟度）、保有資格（取得年月日・有効期限）、現場（プロジェクト）経歴（期間・現場・役割・規模・担当工程・使用スキル）

### 3.3 横断検索・可視化
- 経歴書一覧（REF002）: 全社員の経歴書を検索・一覧表示する。絞り込み条件は氏名・カナ、所属組織、経験年数、スキル（AND/OR）、資格（AND/OR）、**現場（過去にどの現場に誰がいたか）**
- 検索対象は全社員（他部署を含む）。一般社員の場合、他部署の社員は一覧行の表示までとし、経歴書詳細へは遷移できない（ADR `0008`）
- スキルマップ／組織ダッシュボード（REF008）: 組織単位（事業部/部署/Gr）ごとのスキル・資格の保有者数集計と、保有者名の一覧を表示する
- 旧要件の「資格保有者一覧」「スキル保有者一覧」（独立画面）は、REF002の検索条件とREF008の集計表示に統合した（ADR `0007`）

### 3.4 マスタ管理（システム管理者向け）
- アカウント管理（REF007/EDT006/EDT007）: アカウント一覧（氏名あいまい検索、所属組織／権限／状態で絞込み）、新規登録（社員ID・メールアドレス＝Azure AD照合キー・所属部署・初期権限の事前登録）、編集（所属部署・権限の変更、退職・無効化）
- スキルマスタ管理（MST001）: スキルの登録・編集・削除。カテゴリは既存選択／新規入力。バージョンは複数タグ入力
- 資格マスタ管理（MST002）: 資格の登録・編集・削除。カテゴリは既存選択／新規入力（スキルカテゴリとは独立）
- 現場ポジションマスタ管理（MST003）: プロジェクトでの役割（SE/PG/リーダー等）のCRUD
- 部署マスタ管理（MST004）: 組織単位（事業部＞部署＞Gr）の階層の登録・編集・削除。追加ボタンは事業部専用、部署・Grは一覧から配下に追加
- 現場マスタ管理（MST005）: プロジェクト経歴で使用する現場（site）のCRUD ★新規（ADR `0007`）
- 駅・路線のマスタ管理画面は持たない。駅・路線データは初期データ投入スクリプトで整備する（ADR `0007`）

### 3.5 入出力
- 【決定】初回データ投入: 既存Excel台帳からの移行はスクリプトによる直接投入で行う。取込画面は持たない（ADR `0005-csv-import-scope.md`の運用をスクリプト投入に一本化、ADR `0007`）。継続的な一括更新機能としては持たない
- 【決定】一覧のCSV/Excel出力機能は持たない（ADR `0006-remove-export-feature.md`）
- 経歴書のPDF出力: 経歴書詳細・マイページの「PDF出力」ボタンから印刷用プレビュー（REF005）へ遷移し、出力直前の見た目を確認してダウンロードする ★新規（ADR `0007`。一覧出力ではなく個人の経歴書単位の出力のため、ADR `0006`とは対象が異なる）

### 3.6 非機能要件
- セキュリティ: 認証はAzure ADに委譲、権限に応じた画面・操作制御（閲覧制限を含む）を必須とする
- 画面デザイン: 見た目にこだわりたいという要望あり（優先度はフェーズ3）

## 4. フェーズ分け（提案）
| フェーズ | 内容 |
|---|---|
| フェーズ1（MVP・ハッカソンのデモ中心） | ログイン（Entra ID）、トップ、経歴書一覧/詳細（閲覧権限制御含む）、マイページ、基本情報・経歴概要・スキル・資格の登録編集（EDT001〜004）、部署マスタ（階層）CRUD、スキル・資格マスタCRUD |
| フェーズ2 | プロジェクト経歴登録（EDT005）・プロジェクト経歴一覧（REF006）、現場マスタ・現場ポジションマスタ、アカウント管理（REF007/EDT006/EDT007）、印刷用プレビュー（PDF出力）、初回データ投入スクリプト |
| フェーズ3 | スキルマップ／組織ダッシュボード（REF008）、検索・絞り込み強化、画面デザインの作り込み |

## 5. 決定事項・未確定事項

### 5.1 決定済み
| 項目 | 決定内容 | 記録 |
|---|---|---|
| 画面体系 | `01_画面一覧.pdf`（22画面）を正とする。認証はAzure AD（Entra ID）、組織は3階層、現場マスタ新設、PDF出力追加 | ADR `0007-screen-list-pdf-adoption.md` |
| 経歴書の閲覧範囲 | 一般社員は自部署のみ詳細閲覧可。経歴書一覧の検索は全社対象で他部署も結果表示。ロールに人事・営業（HR_SALES）を追加 | ADR `0008-view-permission-department-scope.md` |
| 管理職の編集可能範囲 | 自部署のみ（判定は部署レベルの組織単位一致に更新） | ADR `0002-manager-edit-scope-and-department.md`・`0008` |
| 社員IDの発番ルール | 単純連番（6桁ゼロ埋め、例: `000001`）※既存決定を引き継ぐ | ADR `0003-employee-id-format.md` |
| 論理削除データの一覧表示方法 | 一覧はデフォルト非表示、フィルタ切替で表示可能にする | ADR `0004-soft-delete-list-display.md` |
| 初回データ投入 | スクリプトによる直接投入（取込画面は持たない）。継続運用機能としては持たない | ADR `0005-csv-import-scope.md`・`0007` |
| 一覧のCSV/Excel出力機能 | 持たない（削除）。個人経歴書のPDF出力（REF005）は対象外で、こちらは提供する | ADR `0006-remove-export-feature.md`・`0007` |

### 5.2 未確定事項（要チーム確認）
- 本運用を想定するか、ハッカソン後は学習用として割り切るか。現時点では「学習用MVPとして完成させることを最優先し、本運用に必要な運用体制（監視・バックアップ・SLA等）は今回のスコープ外とする」という前提で設計を進める。本運用を目指す場合は`docs/design/basic-design.md`の非機能要件を見直す必要がある
- Azure ADテナント（開発・デモで使用するテナントとアプリ登録の権限）を誰が用意するか。→ 準備が整うまで開発用ログイン（パスワードなし）で代替することを決定（ADR `0009`）。テナント準備後にEntra IDプロバイダへ切替
- 部署マスタ（事業部＞部署＞Gr）の初期データ（実際の組織名・コード一覧）はテストの時点で管理側が用意する
- 部署異動が発生した場合の履歴管理は行わず、現在の所属のみ保持する前提で進める
- 駅・路線データの追加・修正が必要になった場合の運用（管理画面がないためスクリプト対応となる）
- ~~PDF出力の実現方式~~ → ブラウザ印刷ベースで決定（ADR `0009`）
